Web UI Guide

Complete guide to the Twin2Clouds web interface for configuring and optimizing multi-cloud Digital Twin architectures. This page documents all input parameters, scenario presets, and output interpretation.

Accessing the Interface

After starting the application (see Setup & Usage), open your browser and navigate to: 

http://localhost:8000/

The UI automatically loads pricing data and currency rates on first access.

Scenario Presets

The UI provides three pre-configured scenarios to quickly populate all input fields. Click any preset button to load its values, then customize as needed.

Smart Home

Small scale residential deployment

  • 100 devices
  • 1 message/10 min
  • 1 KB message size
  • 1 month hot storage
  • 10 twins, basic visualization
Industrial Facility

Medium scale manufacturing

  • 4,000 devices
  • 1 message/min
  • 2 KB message size
  • 3 months hot, 6 cool storage
  • 500 twins, 3D visualization
Large Building

Large commercial building

  • 30,000 devices
  • 1 message/5 min
  • 5 KB message size
  • 6 months hot, 12 cool, 24 archive
  • 5,000 twins, complex dashboards

Input Parameters — Core Configuration

These parameters define the fundamental properties of your IoT deployment.

Parameter Type Description Default
numberOfDevices Number Total count of IoT devices/sensors sending data. Affects L1 (messaging) and L4 (twin) costs. 1000
sendingIntervalMinutes Number How often each device sends a message (in minutes). Lower = more messages. 1
averageMessageSizeKB Number Average payload size per message in KB. Affects messaging and storage costs. 1
region Select Deployment region for pricing lookup. Options vary by provider. us-east-1
currency Select Display currency for all cost calculations (USD, EUR, GBP, JPY, CHF). USD

Input Parameters — Storage Retention

Storage costs depend heavily on data retention periods across tiered storage. Use sliders to configure each tier.

Parameter Type Description Range
hotRetentionMonths Slider Hot Storage: Fast, expensive. Data available for immediate queries.
DynamoDB Standard / Cosmos DB / Firestore 		0—24 months
coolRetentionMonths Slider Cool Storage: Balanced cost/performance. Slight access delay.
S3 Standard-IA / Blob Cool / Cloud Storage Nearline 		0—48 months
archiveRetentionMonths Slider Archive Storage: Cheapest, hours to retrieve.
S3 Glacier / Blob Archive / Cloud Storage Archive 		0—120 months
Data Accumulation: Total stored data = messages/month × message_size × retention_months. Long retention with high message volume can significantly impact costs.

Input Parameters — Advanced Settings

Fine-tune layer-specific parameters for precise cost estimation.

Layer 2: Compute
Parameter Type Description Default
processorDurationMs Number Average execution time of processor function in milliseconds. 100
processorMemoryMB Number Memory allocation for processor function (128-3008 MB). 128
Layer 4: Digital Twin
Parameter Type Description Default
numberOfTwins Number Total digital twins to manage. Typically equals or less than device count. 100
twinUpdatesPerMonth Number State updates per twin per month. Often correlates with message volume. 10000
twinQueriesPerMonth Number Read queries against twin state per month. 5000
enable3DModel Checkbox Enable 3D scene/model support (TwinMaker Scene, Azure DT Model). Adds additional compute costs. Unchecked
entityCount Number Number of 3D entities (only shown when 3D model is enabled). 100
Layer 5: Visualization
Parameter Type Description Default
dashboardRefreshPerHour Number Dashboard auto-refresh rate (1-60 times per hour). 12
activeHoursPerDay Number Hours per day the dashboard is actively viewed (affects query costs). 8
editorUsers Number Grafana editor/admin users (higher license cost). 2
viewerUsers Number Grafana viewer users (read-only, lower license cost). 10
apiCallsPerMonth Number External API calls to the visualization layer per month. 10000

Provider Inclusion Toggles

Control which cloud providers are included in the optimization for each layer. Uncheck a provider to exclude it from comparison.

Layer AWS Azure GCP
L1 Data Acquisition Available Available Available
L2 Compute Available Available Available
L3 Storage Available Available Available
L4 Digital Twin Available Available Not Implemented Self-hosted TwinMaker on GCE
L5 Visualization Available Available Not Implemented Self-hosted Grafana on GCE
GCP L4/L5 Status: Self-hosted Digital Twin (TwinMaker) and Visualization (Grafana) on GCP Compute Engine are planned but not yet implemented. The GCP checkboxes for L4 and L5 are disabled in the UI and automatically excluded from calculations.

Interpreting Results

Click Calculate Cost to run the optimization. Results include:

Result Components
Cheapest Path

Visual representation of the optimal provider selection for each layer. Shows the recommended multi-cloud architecture.

Total Monthly Cost

Estimated total monthly cost following the optimal path. Includes cross-cloud transfer costs if path spans multiple providers.

Layer Cost Cards

Flip cards showing cost comparison per layer. Front shows costs, back shows service stack. Click to flip.

Cross-Cloud Transfer Costs

When the optimal architecture spans multiple clouds, additional "glue" costs apply:

  • Egress Traffic: Cost to transfer data out of a cloud provider
  • Connector Function: Serverless function at source to push data
  • Ingestion Function: Serverless function at destination to receive data
Optimization Notes

When cross-cloud transfers are detected, the results section displays Optimization Notes showing the explicit breakdown of transfer costs. This helps you understand if a single-provider solution might be more cost-effective despite higher individual layer costs.
