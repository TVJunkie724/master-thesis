Web UI Guide

A guide to using the Twin2Clouds graphical interface for configuring and optimizing your Digital Twin.

Accessing the Interface

Once the application is running (see Setup & Usage), open your browser and navigate to: http://your-server/ui

Scenario Presets

The UI provides three built-in presets to quickly populate the configuration fields:

Smart Home

Small scale. 100 devices, low data volume, short retention.

Industrial Facility

Medium scale. 4,000 devices, high frequency, longer retention.

Large Building

Large scale. 30,000 devices, 3D models enabled, complex visualization.

Configuration Parameters

Layer 1/3: IoT & Processing

  • Number of Devices: Total count of physical sensors/devices.
  • Sending Interval: How often each device sends a message (in minutes).
  • Message Size: Average payload size in KB.

Layer 2: Storage Retention

Use the sliders to set retention periods for each storage tier:

  • Hot Storage: Expensive, fast access (e.g., 1-3 months).
  • Cool Storage: Cheaper, slower access (e.g., 3-12 months).
  • Archive Storage: Cheapest, very slow access (e.g., 12+ months).

Layer 4: Twin Management

  • 3D Model: Enable if you need a spatial representation (TwinMaker/Azure Digital Twins).
  • Entity Count: Number of 3D objects/nodes in the scene.

Layer 5: Visualization

  • Dashboard Refresh: How often the Grafana dashboard updates per hour.
  • Active Hours: How many hours per day the dashboard is actively viewed.
  • Users: Number of editors (creators) and viewers (read-only).

Interpreting Results

Click Calculate Cost to run the optimization. The results section will display:

  • Optimal Provider per Layer: The cheapest provider found for each architectural layer.
  • Total Monthly Cost: The estimated total cost for the optimized configuration.
  • Cost Breakdown: Detailed costs for AWS, Azure, and GCP for comparison.

Optimization Notes (Glue Code)

If the optimal architecture spans multiple clouds (e.g., AWS IoT → Azure Storage), the UI displays Optimization Notes tables. These highlight the "hidden" costs of cross-cloud integration:

Table Logic

When Provider A (Source) sends data to Provider B (Destination), specific intermediate resources are required:

  • Egress Traffic: Cost to send data out of the source cloud via the internet.
  • Connector Function: A serverless function (Lambda/Azure Function/Cloud Function) at the source to read/push data.
  • Ingestion Function: A serverless function at the destination to receive/write data.

The "Optimization Note" table explicitly lists these add-on costs so you can see why a multi-cloud option might be more expensive than expected.
