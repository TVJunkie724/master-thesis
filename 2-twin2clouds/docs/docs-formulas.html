Cost Formulas & Service Mapping

Mathematical formulas used for dynamic pricing and how each provider's services map to them.

Formula Overview

These formulas are derived from the project's research methodology and standard cloud pricing models. Symbols correspond to service-specific parameters fetched dynamically.

ID Name Expression Description
CM Message-Based c_m * N_m IoT message ingestion (IoT Core, IoT Hub).
CE Execution-Based c_e * max(0, N_e − freeRequests) + c_t * max(0, T_e − freeComputeTime) Serverless compute (Lambda, Functions).
CA Action-Based c_a * N_a API calls, reads/writes, retrievals.
CS Storage-Based c_s * V * D Storage per GB-month (S3, Blob, DynamoDB).
CU User-Based (c_u_editor * N_editor + c_u_viewer * N_viewer) + c_u_hour * H User licensing (Grafana) or VM Time.
CTransfer Data Transfer c_transfer * GB_transferred Egress data transfer.

Parameter Definitions

  • c_m: Price per message
  • N_m: Total messages
  • c_e: Price per execution
  • T_e: Total compute seconds
  • c_s: Price per GB-month
  • V: Storage volume (GB)
  • D: Duration (months)
  • c_u: Price per user/hour
  • H: Total hours
  • c_transfer: Price per GB
  • GB_transferred: Total GB

Formula Mapping Rationale

Serverless compute costs are a function of two distinct variables: the number of requests (invocations) and the duration of execution, which is often weighted by the amount of memory allocated. The CE formula (c_e * N_e + c_t * T_e) accurately models this by summing the request cost (c_e) and the compute-time cost (c_t). It also accommodates free tiers by subtracting free requests and compute seconds from the total usage.

Ingestion Services (IoT Core, IoT Hub, Pub/Sub)

Mapped to: CM (Message-Based) OR CTransfer (Volume-Based)
IoT ingestion services typically charge based on the number of messages processed (AWS IoT Core, Azure IoT Hub), which maps to CM. However, Google Cloud Pub/Sub charges based on data volume (GiB transferred), which maps to CTransfer (c_transfer * GB_transferred). This distinction is crucial for accurate cross-provider comparison.

NoSQL & Document Databases (DynamoDB, Cosmos DB, Firestore)

Mapped to: CA (Action-Based) + CS (Storage-Based)
Modern cloud databases typically charge for two main components: the storage of data at rest and the throughput (reads/writes) used to access it. We map the storage component to CS (c_s * V * D) and the throughput/operation component to CA (c_a * N_a). This separation allows for a granular breakdown of static vs. active costs, which is crucial for optimizing workloads with high storage but low access frequency (or vice versa).

Object Storage (S3, Blob Storage, Cloud Storage)

Mapped to: CS (Storage-Based) [+ CA]
The primary cost driver for object storage is the volume of data stored over time, which is modeled by CS. However, retrieval costs (especially for cooler tiers like Glacier or Archive) and API requests (PUT/GET/LIST) can be significant. These are modeled as additional Action-Based (CA) components to ensure the total cost of ownership is accurate, particularly for archival strategies where retrieval costs can exceed storage savings.

API Gateways & Management

Mapped to: CA (Action-Based) + CTransfer
API gateways are typically charged per API call received, which fits the CA model perfectly. Additionally, data egress fees are often applied at this layer when data leaves the cloud provider's network. These egress costs are captured by the CTransfer formula, ensuring that the cost of serving data to external clients is fully accounted for.

Orchestration & Messaging (Step Functions, Logic Apps, EventBridge)

Mapped to: CA (Action-Based)
These services are billed per discrete unit of work, such as a state transition, a workflow step, or an event published. The Action-Based (CA) formula provides a direct mapping to these usage-based pricing units (e.g., per 1,000 state transitions or per million events), allowing for precise cost estimation based on workflow complexity and event volume.

Self-Hosted Components (VMs / Compute Engine)

Mapped to: CU (User/Time-Based) + CS
Virtual machines incur costs based on running time (hourly or per-second) and attached persistent storage. The CU formula captures the time-based compute cost (c_u_hour * H), effectively treating the "User" variable as a "Time" variable in this context. The CS formula handles the cost of the persistent disk storage attached to the instance.
