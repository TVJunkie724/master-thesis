\section{Motivation}
% Start broad, then narrow to the specific problem.
%
% The need for multi-cloud Digital Twins:
% - Industry 4.0 and IoT are driving adoption of Digital Twins
% - Cloud computing enables scalable DT infrastructure
% - Multi-cloud strategies reduce vendor lock-in, improve resilience,
%   and can optimize costs by selecting best-of-breed services per provider
% - However, deploying DT across multiple clouds is complex:
%   each provider has different APIs, naming, pricing, and resource models
%
% The gap between theory and practice:
% - Academic work on DT architectures (EDT_25, 5-Layer model) provides
%   theoretical frameworks but stops short of actual deployment
% - Cost optimization models exist on paper but are disconnected from
%   the deployment tools that would act on their recommendations
% - No existing tool bridges the gap between "what should I deploy?"
%   (cost optimization) and "deploy it for me" (infrastructure provisioning)

\section{Problem Statement}
% Clearly state what is broken/missing.
%
% Two prior bachelor thesis projects exist:
% - Project A ("The Brain"): a cost calculator for cloud-hosted Digital Twins,
%   limited to hardcoded pricing and monolithic code
% - Project B ("The Muscle"): a Python SDK-based deployer for AWS,
%   limited to a single cloud provider and CLI interaction
%
% These projects cannot be used together because:
% 1. No shared API or data format connects them
% 2. Neither tracks state (no database, no deployment lifecycle)
% 3. Neither supports multi-cloud beyond their original narrow scope
% 4. No user interface exists — both require manual JSON editing
%
% The result: a proof-of-concept gap between theoretical cost optimization
% and practical multi-cloud deployment. Bridging this gap requires
% significant architectural redesign, not just "glue code."

\section{Research Objective}
