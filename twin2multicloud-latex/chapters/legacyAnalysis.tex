\section{Overview of Legacy Prototypes}
\label{sec:legacy-overview}

The Twin2MultiCloud platform builds on two independent bachelor thesis prototypes: a cost optimizer % TODO: CITE --- Gritsch et al., Twin2Clouds (MODELS 2025)
that estimated monthly hosting costs for digital twin deployments, and a cloud deployer % TODO: CITE --- Kaserer, "Developing a Configurable Cloud-Based Digital Twin System with Layered Architecture and Automated Deployment", Bachelor Thesis, University of Innsbruck, 2026
that provisioned the required \gls{aws} infrastructure. Neither project was designed for integration with the other. The optimizer ran entirely in the browser with no server component, while the deployer operated as a local command-line tool. This section describes each prototype's original state, focusing on the architectural decisions and technical constraints that later drove the refactoring effort documented in Sections~\ref{sec:refactoring-optimizer}--\ref{sec:refactoring-management}.

\subsection{Project A: Twin2Clouds (``The Brain'')}
\label{sec:legacy-optimizer}

The Twin2Clouds project, referred to as the Cost Optimizer or ``The Brain'' throughout the rest of this thesis, was a pure client-side JavaScript web application developed as part of a conference paper submission. % TODO: CITE --- Gritsch et al., Twin2Clouds (MODELS 2025)
It had no server component and no backend; users opened an HTML page via any static file server, configured their \gls{iot} deployment scenario through form inputs, and triggered a cost calculation. The result was rendered as HTML cards injected directly into the page's \gls{dom}.

Table~\ref{tab:legacy-optimizer-structure} shows the file layout. The application consisted of nine JavaScript source files containing 35~top-level functions, with no classes (except a utility \texttt{PriorityQueue}), no module system, and no shared interfaces. The central orchestrator, \texttt{cost\_calculation.js}, contained a single 490-line function that handled input reading, validation, all layer cost computations, and HTML output assembly in one monolithic block.

While the optimizer served its purpose as a conference artifact, several architectural limitations prevented it from scaling to a multi-provider production tool:

\textbf{Extensibility.} The application supported only \gls{aws} and Azure. Adding a third provider would have required modifying every layer calculator file, adding new transfer cost functions for each provider pair (the existing 14 would grow to at least 24), and updating the static \texttt{pricing.json} file with new service entries. There was no provider abstraction; each calculator contained provider-specific logic inline.

\textbf{Maintainability.} All pricing data was stored in a static JSON file containing 114~lines of hardcoded values. There was no mechanism to fetch live prices from cloud provider \glspl{api}, so the figures grew stale as soon as any provider adjusted its rates. This is not a theoretical concern: Azure publishes global pricing updates twice per year, % TODO: CITE --- Microsoft, "Global pricing changes for cloud products", 2023
and \gls{aws} has announced over 60 public price reductions since its inception. \gls{gcp} adjusts infrastructure prices quarterly and updates Spot prices as frequently as monthly. With a static pricing file, any of these changes would go unnoticed indefinitely. Furthermore, the Azure serverless compute calculator reused the \gls{aws} Lambda pricing formula verbatim, with a source code comment acknowledging this shortcut. This assumption would silently produce incorrect results if either provider changed their pricing model.

\textbf{Modularity.} The 490-line \texttt{calculateCosts()} function mixed input parsing, validation, computation, and presentation in a single block. A global \texttt{pricing} variable, loaded via \texttt{fetch()} at startup, was accessed by every calculator without encapsulation. The application also used a different layer numbering than the one later adopted in the unified platform: the Twin2Clouds paper defined Layer~2 as Storage and Layer~3 as Processing, following the 5-Layer Architecture reference model, whereas the Cloud Deployer bachelor thesis reordered them so that data flows through layers in sequence (L2=Processing, L3=Storage). The unified platform adopted the deployer's convention, making direct comparison with the original optimizer non-trivial.

\textbf{Testability.} Because every function read directly from the browser's \gls{dom} and the global \texttt{pricing} object, there was no way to run calculations outside a browser context. The project contained no tests.

For the refactored version of the optimizer, see Chapter~\ref{chap:systemArchitecture}, Section~\ref{sec:optimizer}.

\subsection{Project B: Cloud Deployer CLI (``The Muscle'')}
\label{sec:legacy-deployer}

The Cloud Deployer CLI, referred to as the Cloud Deployer or ``The Muscle'' throughout the rest of this thesis, was a Python command-line application that provisioned the entire 5-layer digital twin infrastructure on \gls{aws} through imperative \texttt{boto3} SDK calls. % TODO: CITE --- boto3 (AWS SDK for Python)
There was no \gls{iac} tooling; each resource (IAM roles, Lambda functions, IoT rules, DynamoDB tables, S3 buckets, TwinMaker workspaces, Grafana dashboards) was created by calling the corresponding SDK method directly, with no state file tracking which resources existed. % TODO: CITE --- Terraform (HashiCorp)

The project followed a composition-based architecture. An abstract \texttt{Deployer} base class defined four operations (\texttt{deploy}, \texttt{destroy}, \texttt{info}, \texttt{log}), and 49~concrete deployer classes each handled a single cloud resource. Aggregator classes composed these per-resource deployers into layer-level units that ran sequentially. This structure provided a clear deployment ordering, but it was tightly bound to a single provider: all deployer classes resided under \texttt{deployers/aws/}, and adding Azure or GCP support would have required duplicating the entire directory hierarchy with no shared abstraction.

The deployer's configuration format hinted at multi-provider ambitions that were out of scope of the bachelor thesis. A \texttt{config\_providers.json} file mapped each architectural layer to a cloud provider:

\begin{lstlisting}[language=json, caption={Legacy provider assignment: per-layer configuration with only AWS implemented.}, label={lst:legacy-providers}]
{
  "layer_1_provider": "aws",
  "layer_2_provider": "aws",
  "layer_3_hot_provider": "aws",
  "layer_3_cold_provider": "aws",
  "layer_3_archive_provider": "aws",
  "layer_4_provider": "aws",
  "layer_5_provider": "aws"
}
\end{lstlisting}

Although the schema allowed per-layer provider selection, only \gls{aws} implementations existed. Setting any layer to \texttt{"azure"} would have caused an import error, since no corresponding deployer classes had been written. This gap between the configuration model and the implementation illustrates the prototype's incomplete extensibility.

Beyond the single-provider limitation, several additional architectural constraints affected the system's quality:

\textbf{Coupling.} A central \texttt{globals.py} module (256~lines) served as a shared mutable state container. It held twelve \texttt{boto3} client objects as module-level variables, initialized through functions using Python's \texttt{global} keyword, alongside over thirty resource-naming functions. Every other module imported \texttt{globals} directly. This design meant that any attempt to unit-test a deployer class would transitively trigger all twelve client initializations, which required valid \gls{aws} credentials at import time.

\textbf{Reliability.} If a deployment failed midway, for instance because an IAM role creation timed out, the deployer had no rollback mechanism. Resources created before the failure remained orphaned on \gls{aws}, requiring manual cleanup through the web console. There was no streaming of progress information either; the only feedback during deployment was whatever \texttt{print()} statements individual deployer classes emitted.

\textbf{Security.} Cloud credentials were stored in plaintext in \texttt{config\_credentials.json} on disk, with no encryption and no validation before deployment began.

\textbf{Testability.} The project contained no tests of any kind. The tight coupling through \texttt{globals.py} and the absence of dependency injection made introducing tests prohibitively difficult without first restructuring the entire codebase.

For the refactored version of the deployer, see Chapter~\ref{chap:systemArchitecture}, Section~\ref{sec:deployer}.
