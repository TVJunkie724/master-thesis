\subsection{The SDK-to-Terraform Migration}
% When extending the original AWS SDK deployer to support Azure and GCP,
% the imperative approach collapsed under the weight of cross-provider
% resource dependencies. Manually ordering 50+ resource creation calls
% across three providers was untenable.
%
% The decision to switch to Terraform was not immediate; it came after
% attempting the SDK extension and encountering cascading race conditions.
% This experience validated the "fail-fast, refactor-properly" approach
% described in Chapter~\ref{chap:legacyAnalysis}.
%
% Trade-off: Terraform abstracts away fine-grained control. Some operations
% (e.g., Azure Function Keys, SDK-specific cleanup) still require SDK
% fallback code (the \_run\_sdk\_cleanup pattern in deployer.py).

\subsection{Cross-Cloud Pricing Incomparability}
% Cloud providers intentionally use different units and pricing structures.
% Example: AWS IoT Core charges per message (tiered), while GCP Pub/Sub
% charges per data volume. Comparing "messages" to "bytes" required
% normalization assumptions (average message size).
%
% This challenge directly motivated the Factory + Protocol architecture
% for price fetchers: each fetcher encapsulates provider-specific
% normalization logic behind a common interface.

\subsection{Terraform State Management in Containers}
% Terraform stores state locally by default. In a containerized environment:
% - State files must persist across container restarts (volume mounts)
% - Concurrent deployments of different twins must not conflict (state isolation)
% - Failed deployments leave partial state that must be recoverable
%
% Solution: per-project state files stored in the project upload directory.

\subsection{Cross-Cloud Networking}
% Moving data between providers (e.g., AWS Lambda → GCP Cloud Function)
% requires public endpoints, authentication tokens, and firewall rules.
% This is the most operationally fragile part of the system.
%
% The current webhook-based approach works but adds latency.
% Future work could explore dedicated cross-cloud VPN tunnels or
% event broker intermediaries.
