GCP Deployment Guide

Guide to deploying Digital Twin layers on Google Cloud Platform (in development).

Status: In Development GCP deployment is currently being implemented. Check back soon for updates.

Planned GCP Services

Layer Service Purpose
L1 - Data Acquisition Cloud Pub/Sub Message ingestion and routing
L2 - Data Processing Cloud Functions Dispatcher, processors, persister functions
L3 - Hot Storage Firestore Recent data with low-latency access
L3 - Cool Storage Cloud Storage (Nearline) Historical data accessed occasionally
L3 - Archive Storage Cloud Storage (Archive) Long-term archival data
L3 - Data Access API Gateway External access to hot data (for TwinMaker)
L4 - Twin Management Self-hosted solution Custom digital twin implementation
L5 - Visualization Self-hosted Grafana Dashboards on GKE/Compute Engine
Supporting Cloud IAM Access control and service accounts
Supporting Cloud Scheduler Scheduled cron jobs
Supporting Cloud Workflows Service orchestration

Implementation Roadmap

The GCP deployer implementation follows a modular, layer-by-layer approach, strictly aligned with the AWS architecture:

Layer 1: Data Acquisition (Pub/Sub)

  • Todo Service Account:
    • Create: Dedicated Service Account (e.g., sa-twin-ingest).
    • Permissions: Assign Pub/Sub Publisher role.
  • Todo Topics:
    • Create: iot-telemetry (for sensor data) and iot-events (for system events).
    • Interaction: Decouples ingestion from processing.
  • Todo Subscriptions:
    • Create: Push subscriptions for Cloud Functions (L2).
    • Config: Set acknowledgement deadline to 60s.
  • Todo Dispatcher Function:
    • Trigger: Eventarc triggers from Pub/Sub topics.
    • Single Cloud: Routes incoming telemetry to local processor functions.
    • Multi Cloud: Routes messages to a local Connector Function if the processor is on another cloud.
  • Todo Connector Function [Multi-Cloud Only]:
    • Created: Only if Layer 2 is on a different cloud provider.
    • Logic: Masquerades as a processor function. Wraps the event and sends it via HTTP POST to the remote cloud's Ingestion API.
  • Todo Ingestion Function [Multi-Cloud Only]:
    • Created: Only if Layer 1 is on a different cloud provider.
    • Trigger: HTTP Trigger.
    • Logic: Receives events from a remote Connector and invokes the local Processor function.

Layer 2: Data Processing (Cloud Functions)

  • Todo Gen 2 Functions:
    • Deploy: Python functions using Cloud Run underlying infrastructure.
    • Identity: Use dedicated Service Account with minimal privileges.
  • Todo Persister Function:
    • Single Cloud: Writes cleaned data to Firestore using SDK.
    • Multi Cloud: Sends data via HTTP POST to the remote cloud's Writer API (if L3 is remote).
    • Interaction: Invokes Event Checker function (if enabled).
  • Todo Event Checker Function [Optional]:
    • Trigger: HTTP/Direct Invocation from Persister.
    • Logic: Detects anomalies and triggers alerts.
    • Interaction: Triggers Cloud Workflow (Supporting Services).

Layer 3: Storage & Access (Firestore, GCS, API Gateway)

  • Todo Firestore (Hot):
    • Create: Native mode database for hot telemetry.
    • Collection: Create hot_data collection.
  • Todo Writer Function [Multi-Cloud Only]:
    • Created: Only if Layer 2 is on a different cloud provider.
    • Trigger: HTTP Trigger.
    • Logic: Receives JSON data from a remote Persister and writes it to Firestore.
  • Todo Cloud Storage (Cool/Archive):
    • Create: Buckets for "nearline" and "archive" storage classes.
  • Todo Hot-to-Cold Mover:
    • Type: Cloud Scheduler + Cloud Function.
    • Action: Query old data from Firestore -> Write to nearline bucket -> Delete from Firestore.
  • Todo Cold-to-Archive Mover:
    • Type: Cloud Scheduler + Cloud Function.
    • Action: Move objects from nearline to archive bucket.
  • Todo Data Access (API Gateway) [Multi-Cloud Only]:
    • Create: API Gateway to manage access to Cloud Functions.
    • Action: Expose Hot Reader Function as an API.
    • Purpose: Required only if L4/L5 are on a different cloud or using self-hosted tools that cannot access Firestore natively.

Layer 4: Twin Management (Self-Hosted)

  • Todo Compute Engine:
    • Provision: e2-medium instance for custom twin manager.
  • Todo Container:
    • Deploy: Dockerized twin management service.
    • State: Connect to Firestore for twin state persistence.

Layer 5: Visualization (Self-Hosted Grafana)

  • Todo Deployment:
    • Action: Deploy Grafana OSS container on Compute Engine.
  • Todo Data Sources:
    • Single Cloud: Native Firestore plugin.
    • Multi Cloud: Generic JSON plugin to read from remote API Gateway (if L3 is remote and not Firestore).

Supporting Services

  • Todo Cloud Workflows (Event Workflow) [Optional]:
    • Used in: Layer 2 (Triggered by Event Checker).
    • Create: Workflow definition.
  • Todo Event Feedback Function [Optional]:
    • Used in: Layer 2 (Feedback Loop).
    • Create: Function to send feedback commands to devices (via Pub/Sub).
  • Todo Hot Reader Function [Multi-Cloud Only]:
    • Used in: Layer 3 (Data Access).
    • Create: Function to read data from Firestore for API Gateway.
  • Todo Error Handling System [Optional]:
    • Cloud Pub/Sub (Error Topic):
      • Topic: {digital_twin_name}-errors.
      • Purpose: Centralized error routing.
    • Error Reporter Function:
      • Trigger: Pub/Sub Trigger.
      • Logic: Formats error and writes to Firestore Error Collection.
    • Error Collection (Firestore):
      • Collection: system_errors.
      • Purpose: Persistent storage for system errors (Cloud-Side Persistence).

Technical Specifications

Cross-Cloud Implementation Details

  • Configuration & Discovery:
    • Mechanism: Environment Variables.
    • Variables: REMOTE_INGESTION_URL, REMOTE_WRITER_URL, REMOTE_READER_URL.
  • Authentication (Public Endpoints):
    • Mechanism: OIDC Tokens (Service Account Identity).
    • Ingestion/Writer API: Protected via Cloud IAM (invoker role).
  • Error Handling Strategy:
    • Pipeline: Components publish to Cloud Pub/Sub.
    • Persistence: Reporter writes to Firestore (Cloud-First).
    • Reliability: Connectors implement exponential backoff retries. Dead Letter Topics configured.
  • Data Contracts:
    • Format: Standard JSON Envelope 
      { 
    "source": "gcp", 
    "layer": "L2", 
    "payload": {...},
    "timestamp": "2023-10-27T10:00:00Z"
}
