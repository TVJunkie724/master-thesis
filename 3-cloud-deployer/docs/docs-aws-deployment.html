AWS Deployment Guide

Complete guide to deploying Digital Twin layers on Amazon Web Services (AWS).

Status: Fully Implemented AWS deployment is fully operational for all 5 layers.

AWS Services Used

Layer Service Purpose
L1 - Data Acquisition AWS IoT Core Device connectivity, message routing
L2 - Data Processing AWS Lambda Dispatcher, processors, persister functions
L3 - Hot Storage DynamoDB Recent data with low-latency access
L3 - Cool Storage S3 Infrequent Access Historical data accessed occasionally
L3 - Archive Storage S3 Glacier Deep Archive Long-term archival data
L3 - Data Access Amazon API Gateway External access to hot data (for TwinMaker)
L4 - Twin Management AWS IoT TwinMaker 3D digital twin models and queries
L5 - Visualization Amazon Managed Grafana Dashboards and monitoring
Supporting IAM Access control, roles, policies
Supporting EventBridge Scheduled data movers
Supporting Step Functions Event workflow orchestration

Implementation Roadmap

The AWS deployer implementation follows a modular, layer-by-layer approach:

Layer 1: Data Acquisition (IoT Core)

  • Dispatcher IAM Role:
    • Created: {digital_twin_name}-dispatcher-role
    • Purpose: Defines permissions for the entry-point Lambda function.
    • Policies: Attaches AWSLambdaBasicExecutionRole for logging and AWSLambdaRole for invoking other functions.
  • Dispatcher Function:
    • Runtime: Python 3.13.
    • Logic: Acts as the central router, inspecting incoming MQTT messages and routing them to specific processor functions based on topic or payload.
    • Interaction: Decouples ingestion from processing, allowing for scalable message handling.
  • IoT Rule:
    • SQL: SELECT * FROM '{digital_twin_name}/iot-data'.
    • Action: Triggers the Dispatcher Lambda for every message published to the topic.
    • Permission: Explicitly grants IoT Core permission to invoke the specific Lambda function.

Layer 2: Data Processing (Lambda & Step Functions)

  • Persister Function:
    • Role: {digital_twin_name}-persister-role (DynamoDB Full Access).
    • Logic: Validates and cleans raw telemetry data before writing it to the hot storage layer (DynamoDB).
    • Interaction: Asynchronously invokes the Event Checker to screen for anomalies without blocking the ingestion pipeline.
  • Event Checker Function:
    • Role: {digital_twin_name}-event-checker-role (Step Functions Access).
    • Logic: Evaluates data against defined thresholds (e.g., temperature > 80°C).
    • Interaction: If an anomaly is detected, it triggers the Step Function workflow to handle the event.
  • Event Workflow (Step Functions):
    • Type: Standard State Machine.
    • Definition: Orchestrates a multi-step remediation process (e.g., LambdaA -> LambdaB).
    • Purpose: Ensures reliable execution of complex business logic with built-in retries and error handling.
  • Event Feedback Function:
    • Role: {digital_twin_name}-event-feedback-role (IoT Data Access).
    • Logic: Closes the control loop by publishing commands back to the device via MQTT (e.g., "Shutdown").

Layer 3: Storage & Access (DynamoDB, S3, API Gateway)

  • DynamoDB (Hot):
    • Table: {digital_twin_name}-hot-iot-data.
    • Purpose: Stores high-frequency, low-latency data for real-time dashboards.
    • Schema: Uses deviceId as Partition Key and timestamp as Sort Key for efficient time-series queries.
  • S3 Buckets (Cool/Archive):
    • Cool: {digital_twin_name}-cool-storage (Standard-IA) for data accessed monthly.
    • Archive: {digital_twin_name}-archive-storage (Glacier Deep Archive) for long-term compliance retention.
  • Hot-to-Cold Mover:
    • Type: Scheduled Lambda (EventBridge Rule).
    • Schedule: Runs daily (cron).
    • Logic: Queries data older than 7 days from DynamoDB, batches it to S3 Cool, and deletes it from the hot table to manage costs.
  • Cold-to-Archive Mover:
    • Type: Scheduled Lambda (EventBridge Rule).
    • Schedule: Runs daily (cron).
    • Logic: Transitions objects older than 90 days from S3 Cool to Glacier Deep Archive for lowest-cost storage.
  • Data Access (API Gateway):
    • API: REST API exposing the Hot Reader Lambda.
    • Purpose: Provides a standardized interface for external consumers (like TwinMaker or custom apps) to query hot data securely.

Layer 4: Twin Management (IoT TwinMaker)

  • Workspace:
    • Created: {digital_twin_name} workspace.
    • Role: {digital_twin_name}-twinmaker-role.
    • Purpose: The central container for all digital twin resources, including entities, scenes, and resource libraries.
  • Entities & Components:
    • Entities: Virtual representations of physical devices in a hierarchy.
    • Components: Data connectors that bind entities to their data sources (e.g., mapping a "Temperature" property to the API Gateway endpoint).

Layer 5: Visualization (Managed Grafana)

  • Workspace:
    • Created: {digital_twin_name}-grafana.
    • Auth: Integrated with AWS SSO / SAML for secure user access.
    • Role: Grants Grafana permission to read from TwinMaker and S3.
  • Dashboards:
    • Source: Uses the AWS IoT TwinMaker plugin.
    • Visualization: Combines 3D Scene Viewers (spatial context) with Time Series panels (temporal data) for a complete operational view.

Supporting Services

  • IAM (Access Control):
    • Used in: All Layers.
    • Implementation: Creates granular, least-privilege roles for every service to ensure secure service-to-service communication.
  • EventBridge (Scheduler):
    • Used in: Layer 3 (Lifecycle Management).
    • Implementation: Manages cron schedules for data mover functions, ensuring automated data lifecycle transitions.
  • Step Functions (Orchestration):
    • Used in: Layer 2 (Event Workflow).
    • Implementation: Provides state management and visual workflows for complex event handling logic, superior to chaining Lambda functions directly.

Quick Deployment

# Deploy all layers to AWS
curl -X POST "http://localhost:8000/deploy?provider=aws"

Full deployment typically takes 5-15 minutes depending on the number of IoT devices and complexity.

Layer-by-Layer Deployment

For better control and debugging, deploy each layer individually:

# Layer 1: IoT Dispatcher
curl -X POST "http://localhost:8000/deploy_l1?provider=aws"

# Layer 2: Persister/Processor
curl -X POST "http://localhost:8000/deploy_l2?provider=aws"

# Layer 3: Storage (Hot, Cool, Archive)
curl -X POST "http://localhost:8000/deploy_l3?provider=aws"

# Layer 4: TwinMaker
curl -X POST "http://localhost:8000/deploy_l4?provider=aws"

# Layer 5: Grafana
curl -X POST "http://localhost:8000/deploy_l5?provider=aws"

Deployment Verification

After deployment, verify resources in AWS Console:

  1. IoT Core: Check Things, Certificates, Policies, and Rules
  2. Lambda: Verify functions starting with {digital_twin_name}-
  3. DynamoDB: Check table {digital_twin_name}-hot-iot-data
  4. S3: Verify buckets for cold, archive, and TwinMaker
  5. TwinMaker: Check workspace and entities
  6. Grafana: Access workspace URL from console

Or use the API:

curl "http://localhost:8000/check?provider=aws"

AWS-Specific Troubleshooting

  • IAM propagation delays: Wait 30-60 seconds after creating IAM roles
  • DynamoDB table not deleting: Tables can take several minutes to delete
  • S3 bucket deletion fails: Buckets must be empty. The deployer empties them automatically.
  • Lambda deployment package too large: Keep dependencies minimal in Lambda functions
  • TwinMaker workspace creation fails: Ensure S3 bucket is created first
